## 1. 产品概述

### 1.1 产品定位
AWE 是一款面向桌面端与本地自动化场景的 **Agentic RPA 基础设施**[cite: 2]，核心主张"AI 编排、人类管理"：由大模型基于结构化的节点字典动态编排自动化工作流，人类负责审查关键节点和最终确认[cite: 2]。

### 1.2 目标用户与协作场景
- **目标受众**：具备一定技术基础、会编写基础 Python/RPA 脚本的开发者或超级个体。系统不追求对“零基础小白”的过度 UI 封装，主打硬核、透明、极简的开发体验。
- **协作模式（架构预留）**：v3.0 专注于单机单人使用。但在底层 SQLite 的 `workflows` 表中，预留 `workspace_id`（租户空间）、`version`（乐观锁防冲突）、`status`（草稿/发布状态）等字段，为未来企业版的多人异步协作编辑打下架构基础。

### 1.3 核心价值主张
- AI 自动将自然语言意图编译为可执行的工作流图，并做编译期静态校验，保证图谱可运行[cite: 2]。
- 深度复用现成开源项目（画布引擎、图编排引擎、RPA 驱动），降低自研成本[cite: 2]。
- "静默按需下载 + CDP 动态接管"的隔离式部署架构，降低分发和环境依赖成本[cite: 2]。

### 1.4 商业模式与产品策略
- **开源内核模式 (Open-Core)**：项目采用“开源单机核心 + 闭源团队协作”的商业路径。
- **免费社区版**：开源核心代码（建议采用防御性开源协议如 AGPL 或 Fair-Code）。包含所有基础原子节点、单机 SQLite 存储、DrissionPage 底层驱动与完整的 AI 编排能力。主打个人开发者与极客的“单机效率神器”。
- **商业增值版（规划中）**：面向企业与团队，提供闭源的高级能力订阅。包括但不限于：多人异步协作、角色权限管理、SSO 单点登录、以及长程任务的分布式集群调度。

## 2. 产品架构与运行模式

| 模式 | 说明 |
|------|------|
| **Mode A（独立桌面端）** | FastAPI + PyWebview 桌面应用，彻底分离运行环境依赖[cite: 2] |
| **Mode B（无头 MCP 服务）** | 作为标准 MCP Server，暴露内部图谱与节点字典供外部控制[cite: 2] |
| **Mode C（后台静默调度）** | 基于 APScheduler + SQLite 的跨日长程任务计划调度[cite: 2] |

## 3. 技术选型（不造轮子清单）

系统严禁从零手写基础轮子，强制集成以下现成开源项目[cite: 2]：

### 3.1 前端
- **可视化画布引擎**：字节跳动开源 `@flowgram.ai`，承载节点渲染、拖拽连线、小地图[cite: 2]
- **框架与样式**：`React 18` + `Vite`；样式库 `shadcn/ui` + `TailwindCSS v4`（复制粘贴源码方式）[cite: 2]

### 3.2 后端
- **Web 与打包框架**：`FastAPI`（异步 API）+ `PyWebview`（桌面窗口封装）+ `PyInstaller`（打包）[cite: 2]
- **图编排与状态机**：`LangGraph`，托管多线并行、条件路由循环、全局状态字典[cite: 2]
- **RPA 自动化驱动**：
  - 网页控制：`DrissionPage`（接管 CDP 协议）[cite: 2]
  - 桌面控制：`pywinauto`（UIA 解析）+ `PyAutoGUI`（图像匹配视觉辅助）[cite: 2]
- **模型调用与知识库**：`LiteLLM`（统一适配各大模型 API）+ `ChromaDB`（单机向量库）[cite: 2]

## 4. 核心功能规格

### 4.1 主界面（工作流列表）
**布局**：两栏布局[cite: 2]。
| 区域 | 宽度 | 内容 |
|------|------|------|
| 左栏 | 固定 240px | Logo + 全局导航（工作流列表 / 节点管理 / 执行历史 / 回收站 / 设置）+ 后端健康状态点 + 底部版本号[cite: 2] |
| 右栏 | 自适应 | 当前页面内容[cite: 2] |

**工作流列表交互规则**：
- 单击名称：仅选中[cite: 2]。
- 双击名称：进入编辑器[cite: 2]。
- Ctrl/Cmd / Shift + 点击：多选切换或范围选择[cite: 2]。

**回收站与软删除机制**：
- **软删除**：按下 Delete 键或右键删除时，不再执行物理删除。数据库对应记录的 `status` 字段标记为 `deleted`，并从主列表中隐藏。
- **回收站入口**：左侧导航栏新增“回收站”入口，展示所有已软删除的工作流，提供“一键还原”和“彻底删除（物理 DELETE）”功能。

### 4.2 工作流编辑器
**布局**：全屏编辑模式（进入后隐藏左侧导航栏）[cite: 2]。
- **节点渲染**：节点主视图直接显示类型色块 + 名称 + **内嵌 config 摘要预览**。节点面板（添加节点时）hover 弹出简介浮卡[cite: 2]。

**撤销/重做 (Undo/Redo) 机制**：
- **作用域**：仅记录图拓扑结构的变化（节点的增/删、连线的增/删、画布位置拖动）。
- **历史栈上限**：最多保留 20 步。
- **生命周期**：内存级维护。点击顶栏「← 返回」离开编辑器，或点击「发版」后，历史栈立刻清空。

### 4.3 节点定义与目录
系统内置 12 大原子节点，分 5 大类：触发与边界、高级 AI 与语义路由、本地知识与数据持久化、外部生态与控制、人类管理[cite: 2]。

**节点 Schema 兼容与自动升级策略**：
为应对后续节点参数版本的迭代，免除人工迁移，系统采用双场景自动升级机制：
- **场景 1（运行时 AI 动态升级）**：执行触发时，若检测到旧版节点，引擎将旧数据与新 Schema 喂给 AI，由 AI 自动完成参数补齐对齐后再投入执行。
- **场景 2（编辑期静默升级）**：用户双击进入编辑器时，前端在加载图谱时自动遍历节点，后台静默合并默认值（补齐新增、剔除废弃），用户点击“发版”即可完成最新 Schema 落盘。

### 4.4 执行历史页面 (HistoryPage)
- **基础展示（v3.0）**：列表展示工作流名称、开始时间、总耗时、最终状态（成功/失败）。点击单条记录，侧边栏直接输出原生文本日志（包含 Python Traceback 报错、HTTP 状态码等），供开发者硬核排查。
- **预留升级**：未来可升级为图形化的执行轨迹重放、耗时卡片统计等。

### 4.5 设置页面 (SettingsPage)
- **基础展示（v3.0）**：前端/后端/桌面壳版本号展示、后端连接状态及当前 CDP 端口。提供 LiteLLM 所需的通用 API Key 录入表单。
- **预留升级**：未来支持并发数配置等功能。

### 4.6 浏览器操作录制工具
1. 编辑器内新增“录制”入口（底部左工具栏）。
2. 复用 Mode A 已建立的 CDP 会话（DrissionPage 调试端口），不引入独立浏览器实例。
3. **多标签页处理**：录制器仅如实记录 `target="_blank"` 的点击，由 AI 编排时利用 DrissionPage 原生的 `page.get_tab()` 语法自动生成代码。
4. **敏感数据处理**：录制时对密码等敏感输入不做脱敏，直接捕获真实值写入生成的脚本中，保证本地使用的极致效率。
5. **执行与确认**：AI 生成的自动化代码将作为 `Skill_Python` 节点插入画布。不强制介入复杂的人工审批流，用户肉眼检查代码无误后，自行点击底部的“运行”按钮即为确认。

## 5. AI 编排引擎规格
- **节点说明书注入**：所有基础节点的 JSON Schema 浓缩为"节点字典"注入 System Prompt[cite: 2]。
- **编译期静态拓扑校验与自愈**：在内存中假跑校验数据映射链条[cite: 2]。

## 6. UI/UX 设计规范
- **主题风格**：严格遵循 `shadcn/ui` 默认设计体系：白色背景 + 深黑文字，零渐变、零紫色、纯扁平[cite: 2]。

## 7. 非功能性需求边界设定
- **AI 编排自愈上限**：拓扑校验打回重写最多允许重试 **3 次**。超时直接截断并在图谱报错，转交人工介入处理，控制 Token 成本。
- **并发浏览器会话控制**：为兼顾本地内存，系统默认最大并发执行数设为 **5**。超出的任务将自动进入队列排队等待。
- **长程任务与崩溃恢复**：v3.0 暂不处理意外关机或进程崩溃导致的任务中断恢复，按丢弃处理。

## 8. 安全与合规
- **8.1 Skill_Python 沙盒加固**：弃用简单的 AST 黑名单，采用 Python 3.8+ 内置的 `sys.addaudithook` 机制从底层拦截高危操作（如 os.system、敏感网络请求）。利用 `exec()` 配合干净的 `globals()` 和 `locals()` 阻断反射逃逸。
- **8.2 Mode B 鉴权与边界**：作为无头 MCP Server 暴露时，服务端口强制且仅绑定 `127.0.0.1`，物理阻断局域网及公网访问。因严格物理隔离，无需额外 API Key 鉴权。
- **8.3 CDP 端口安全**：启动 Chromium 强制附加 `--remote-debugging-address=127.0.0.1`。
- **8.4 密钥管理**：用户录入的大模型 API Key 采用简单对称加密（如 AES / Fernet）后存入 SQLite。主密钥（Master Key）于首次启动时随机生成并独立存入 `.env` 文件，实现密钥与数据的分离存储。

## 9. 数据管理
- **存储方案**：工作流定义、执行记录均基于单文件 SQLite[cite: 2]。
- **极简数据备份策略**：每次运行 `start.bat` 启动系统时，Python 脚本自动将 `data.db` 拷贝并重命名为 `data_backup_YYYYMMDD.db` 放入 `backups` 文件夹。默认保留最近 7 天的备份，防备数据库损坏。

## 10. 工程与发布管理
- **启动方式**：默认 `start.bat` 启动后端 + PyWebview 独立桌面窗口。支持 `--dev` 开启 Vite HMR 热更，支持 `--skip-port-clean` 跳过端口清理[cite: 2]。

## 11. 测试与验收标准
- **黄金链路验证**：由于单机开发精力受限，不强求高覆盖率单元测试。验收唯一基准为“核心工作流的顺畅闭环”：给定一个自然语言指令（如：“搜索知乎关于AI的新闻保存到表格”），必须顺畅走完【生成图谱 -> 拓扑校验自愈通过 -> 本地运行成功 -> 查看到结果日志】全链路，即视为达到发版标准。
